  - `0.000` (drop=0.0)
  - `0.405` (drop=0.4)

  - `78` (drop=0.0)

- Метрики канала/стека (MetricSupervisor) почти не меняются:
  - PRR: `0.9529 -> 0.9518`
  - latency: `~12.0 ms`

- Safety proxy по netstate в этой топологии не показал аварийной зоны:
  - `min_gap_m = 6.01`
  - `min_ttc_s = 3.67..4.44`
  - `risky_gap_events = 0`, `risky_ttc_events = 0` (при порогах 2.0 м / 1.5 с)

## Интерпретация для цели исследования

1. Потери CAM уже в этом сценарии доказуемо влияют на поведение: реакций становится меньше в 5+ раз (`78 -> 15`).
2. Средние сетевые KPI при этом могут выглядеть почти неизменными; значит, для CAV/AV надо контролировать не только средний PRR/latency, но и факт пропуска критических сообщений на уровне приложения.
3. Для демонстрации именно «потеря сообщения привела к аварии» нужно усилить конфликтность сцены (stalled vehicle / blocked lane / tighter time headway), потому что текущая карта и плотность дают безопасные зазоры даже при высоких потерях.
